# 📋 План тестирования Docling Pipeline → RAG-ready Output

## Цель

Убедиться, что выходные файлы (`.md` + `.json` + `chunks.json`) точно отражают содержимое оригинальных PDF **без потерь** в структуре текста, таблиц и изображений/чертежей — и что пайплайн готов для масштабирования на весь набор документов.

---

## 1. Типы тестовых документов

Каждый тип покрывается одним PDF-файлом, по 1 репрезентативной странице:

| # | Тип документа | Файл | Что проверяем |
|---|---|---|---|
| T1 | Таблица (vector outline) | `table11.pdf` | Извлечение числовых ячеек, структура строк/столбцов |
| T2 | Чертёж + схема | `chert+schema.pdf` | Наличие изображения, bbox, отсутствие фантомного текста |
| T3 | Смешанная страница (схема + таблица) | `shema+table.pdf` | Разделение типов: text vs table vs image |
| T4 | Чертёж без таблиц | `chert-1.pdf` | Ни одного чанка type=table, все изображения экспортированы |

---

## 2. Уровни проверки

### Уровень 1 — Структура JSON-чанков

Проверяется каждый чанк в `{doc}_chunks.json`.

**Метрики:**
- `page` ≠ null для всех чанков
- `bbox` ≠ null для всех чанков
- `type` ∈ {text, table, picture, caption, section_header, list_item}
- `content` ≠ "" (если тип text/table)
- Если `type=picture` → `image_path` не null и файл PNG существует

**Команда запуска:**
```bash
cd /home/art/CLaude/projects/volhovec/src
source venv_docling/bin/activate
python run_docling_test.py
python export_chunks.py output/table11.json
python export_chunks.py output/shema+table.json
```

**Критерий приёмки:** Нет ни одного чанка с `page: null` или пустым `bbox`.

---

### Уровень 2 — Полнота таблиц

Для файлов T1 и T3 сделать визуальное сравнение.

**Алгоритм:**
1. Открыть оригинальный PDF в просмотрщике (страница 1).
2. Открыть соответствующий `{doc}.md`.
3. Пересчитать строки таблицы в PDF вручную → сравнить с числом строк в Markdown-таблице.

**Контрольные числа для `table11.pdf`:**
- Ожидаемое число строк размерного ряда: **~30+ уникальных значений h\*** 
- Ожидается повторяющийся заголовок "Размер полотна" (merged cells) ✓

**Метрика:** `строк в MD ≥ 90% от строк в PDF`

---

### Уровень 3 — Сохранность текста (кириллица)

Открыть `{doc}.md` для типов T1/T3.

**Проверяем:**
- Нет "кракозябров" типа `Pa3MepbIyKa3aHbInpW`
- Цифры не перепутаны: `006` должно быть как `900`, **или хотя бы регулярно** (`006 = 006` в том же контексте)
- Текстовые блоки (подписи, примечания) читаемы и осмысленны

**Команда:**
```bash
grep -i "Pa3" output/table11.md      # не должно быть результатов
grep -i "кракозябр\|Pa3\|Bbi" output/*.md
```

---

### Уровень 4 — Полнота изображений

**Для T2 (чертёж) и T3 (смешанный) сравнить:**
- Число изображений в `chunks.json` (тип=picture) ↔ число видимых изображений в PDF
- Все `image_path` указывают на реально существующие PNG-файлы

**Команда:**
```bash
python - <<'EOF'
import json, os
for doc in ["table11", "shema+table", "chert+schema"]:
    with open(f"output/{doc}_chunks.json") as f:
        chunks = json.load(f)
    images = [c for c in chunks if c["type"] == "picture"]
    missing = [c for c in images if c["image_path"] and not os.path.exists(c["image_path"])]
    print(f"{doc}: {len(images)} изображений, {len(missing)} с ошибкой пути")
EOF
```

**Критерий приёмки:** `missing == 0` для всех документов.

---

### Уровень 5 — Финальный RAG-тест (сквозной)

Имитируем запрос к RAG-системе вручную:

1. Открыть `table11_chunks.json`.
2. Найти чанк с текстом "2000" (число из таблицы размеров).
3. Убедиться, что у этого чанка:
   - `page` указывает корректно (страница 1)
   - `bbox` покрывает разумную область (x2 > x1, y2 < y1 в PDF-координатах)
4. Открыть PDF на указанной странице и визуально подтвердить соответствие.

## 4. Критерии масштабирования (Go/No-Go для RAG)

| Критерий | Порог |
|---|---|
| Пустых bbox | 0% |
| Изображений с ошибкой пути | 0% |
| Кириллический OCR (визуально) | >95% разборчивости |
| Полнота строк таблицы | ≥90% от оригинала |
| Чанки с page: null | 0% |

Если все 5 критериев прошли — пайплайн **готов к индексации в векторную БД**.
