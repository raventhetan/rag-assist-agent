# 📘 Техническое задание: Docling Table-Native OCR Pipeline с RAG-поддержкой (MVP)

---

## 1️⃣ Цель проекта
Построить пайплайн конвертации сложных технических документов (PDF: чертежи, прайс-листы, схемы; Excel прайс-листы) в структурированный `RAG-ready` JSON-формат для последующей загрузки в базу знаний LLM (чат-бота). 

Система должна уметь:
1. Сохранять `Markdown`/`JSON` с максимально точной структурой таблиц.
2. Безошибочно распознавать **кириллицу** даже на растровых и "зашитых" в вектор таблицах.
3. Сохранять пространственную информацию о каждом текстовом/табличном блоке (`bbox`, `страница`).
4. Автоматически извлекать и сохранять **изображения / схемы** в виде отдельных `PNG`-файлов и делать на них ссылку в результирующем `JSON`. Это необходимо, чтобы чат-бот мог отвечать пользователю не только текстом, но и картинкой-чертежом, а также давать точную ссылку на источник ответа ("Страница 5, таблица 2").

---

## 2️⃣ Архитектурный выбор (Ключевая находка)
После серии тестов выяснилось, что чистый Tesseract уничтожает табличную вёрстку Docling, а нативный гибридный пайплайн (PaddleOCR + PyMuPDF) избыточен. 

**Финальное решение:** 
Единый конвертер — библиотека `Docling`, использующая встроенный `TableFormer` для поиска сетки таблиц (вектор + растр) и перенастроенная на внешнюю кастомную модель `PP-OCRv5 (Cyrillic)` от фреймворка `PaddleOCR` для идеального распознавания русских букв.

### Преимущества:
- Docling сам читает лёгкие текстовые PDF и Excel без вызова OCR.
- На тяжелых файлах работает `force_full_page_ocr=True`. Сетка таблиц не ломается.
- Кастомная модель `PP-OCRv5` идеально читает "кракозябры" и сложные шрифты на чертежах.
- Docling изначально поддерживает экспорт `bbox`, страниц и `PNG`-кадров из документа.

## 3️⃣ Рекомендуемый стек технологий

| Задача | Библиотека / Решение | Обоснование |
|---|---|---|
| **Pipeline Runner** | `Python 3.10` + `Docling 2.x` | Оркестратор конвертации и парсер структуры. |
| **Grid / Table Detection** | Встроенный `TableFormer` (Docling) | Идеально распознает слепые векторные таблицы. |
| **OCR Backend** | `RapidOcrOptions` (движок ONNX) | Позволяет подменить стандартные модели. |

| **Hardware / Acceleration** | `CUDA 11.8` (GTX 1050 Max-Q) | Обработка страницы ~30 сек с максимальным OCR. |

## 5️⃣ Описание пайплайна (Stages)

Основой служит `docling_eval/run_docling_test.py`.

### Stage 1: Конфигурация Docling (Custom RapidOCR)
Мы создаем инстанс конвертера, пробрасывая пути к нашим загруженным кириллическим моделям.
```python
from docling.datamodel.pipeline_options import RapidOcrOptions, ThreadedPdfPipelineOptions, TableStructureOptions

pipeline_options = ThreadedPdfPipelineOptions(
    do_ocr=True,
    do_table_structure=True,
    images_scale=2.5, # Улучшенное разрешение для OCR и картинок
    ocr_options=RapidOcrOptions(
        force_full_page_ocr=True,
        rec_model_path="models/cyrillic/cyrillic_PP-OCRv5_rec_mobile_infer.onnx",
        rec_keys_path="models/cyrillic/ppocrv5_cyrillic_dict.txt"
    ),
    table_structure_options=TableStructureOptions(
        do_cell_matching=True,
        mode="accurate"
    )
)
```

### Stage 2: Настройка экспорта изображений (Picture Extraction)
Чтобы извлечь схемы чертежей в высоком качестве в виде PNG:
В `ThreadedPdfPipelineOptions` параметров документа необходимо активировать извлечение картинок (через `PdfFormatOption(pipeline_options=..., picture_options=...)` и дальнейший вызов функции сохранения картинок из объекта Docling). Изображения должны сохраняться в структуру `output/images/{doc_stem}/`.

### Stage 3: Извлечение и парсинг
1. Скрипт прогоняет `PDF` или `XLSX` через Docling. 
2. На выходе получается внутренний `result.document` объект.

### Stage 4: Итоговый экспорт (Spatial-Aware Format)
Это самая важная часть. Для дальнейшего RAG нам необходимо сохранить целостность документа (страничность, абзацы). Мы не занимаемся сейчас смысловым "нарезанием/чанкованием", мы просто экспортируем Docling-сегменты один-в-один, но в плоском удобном `JSON` массиве узлов (nodes).

Формат целевого `nodes.json` (один файл на каждый исходный PDF):
```json
[
  {
    "id": "node_001",
    "type": "text",
    "page": 1,
    "bbox": [10.5, 20.1, 150.0, 30.5],
    "content": "РАЗМЕРЫ ПО ШИРИНЕ",
    "image_path": null
  },
  {
    "id": "node_002",
    "type": "image",
    "page": 1,
    "bbox": [50.0, 100.0, 400.0, 300.0],
    "content": null,
    "image_path": "output/images/table11/page_1_img_1.png"
  },
  {
    "id": "node_003",
    "type": "table",
    "page": 1,
    "bbox": [10.0, 350.0, 800.0, 900.0],
    "content": "| Размер | Световой проход | ...\n| ... | ... | ...",
    "image_path": null
  }
]
```
Эти узлы сохраняют точный тип блока (абзац, таблица, картинка) и его геометрию в документе, не ломая структуру.

## 7️⃣ Сценарий использования (В контексте RAG)

1. Новый файл кладут в систему. 
2. Отрабатывает `Docling pipeline` (извлечение Markdown, PNG-картинок).
3. `Post-processor` генерирует плоский `nodes.json`.
4. В будущем `RAG векторная БД` делит текстовые ноды на смысловые фрагменты (если нужно).
5. При ответе LLM выдаёт текст, берет из векторной БД ссылку на исходный `bounding box` и страницу, и, если контекстом для ответа послужили картинки (чертежи) — отправляет пользователю в чат абсолютный путь до `image_path` PNG-файла.
